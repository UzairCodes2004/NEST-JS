# CHANGELOG

This document details the issues found with the Prisma setup in this NestJS + MySQL project and the changes made to resolve them.

## 1. What Was Wrong

1. **Incorrect Namespace Instantiation & Trailing Comma (Syntax Error)**:
   - In `src/users/users.service.ts`, the code imported `Prisma` from the client and attempted to instantiate it: `const prisma = new Prisma(),`.
   - **Issues**:
     - `Prisma` is a type/namespace generated by Prisma Client, not a constructable class (hence the error *"This expression is not constructable. Type 'typeof import(.../generated/prisma/internal/prismaNamespace)' has no construct signatures."*). The actual constructable class is `PrismaClient`.
     - The line ended with a comma `,` instead of a semicolon `;` right before the `@Injectable()` decorator, creating a parsing syntax error.
2. **Incorrect Module Resolution and Generator Pathing**:
   - `schema.prisma` contained a custom `output = "../src/generated/prisma"` definition.
   - `tsconfig.json` was configured to use `"module": "nodenext"` and `"moduleResolution": "nodenext"`.
   - **Issues**:
     - Because `package.json` does not set `"type": "module"`, TypeScript compiled the custom generated client files as CommonJS, but left ESM-native `import.meta.url` untouched. Node.js failed to execute this combination, throwing `ReferenceError: exports is not defined in ES module scope` at runtime.
     - NodeNext configurations are often incompatible with NestJS's default CommonJS builder CLI out of the box.
3. **Improper Lifecycle Method Call**:
   - In `src/database/database.service.ts`, the NestJS lifecycle hook `onModuleInit` was doing `await this.$connect` instead of calling it as a function: `await this.$connect()`.
4. **Environment Variables Loading Issue**:
   - The application was not loading `.env` variables at runtime when started via Nest CLI, resulting in `DATABASE_URL environment variable is missing` since Prisma 7 connection details are handled inside application code.

---

## 2. What We Changed

1. **Configured `tsconfig.json` for CommonJS**:
   - Changed `"module"` to `"commonjs"`.
   - Changed `"moduleResolution"` to `"node"`.
   - Removed `"resolvePackageJsonExports"` (which requires `node16`, `nodenext`, or `bundler`).
2. **Removed Custom Output from `schema.prisma`**:
   - Removed `output = "../src/generated/prisma"` from the generator block so that the Prisma Client is generated to the default path: `node_modules/@prisma/client`.
   - Removed `url = env("DATABASE_URL")` from the `schema.prisma` `datasource` block for compatibility with Prisma 7 (which moves connection details to `prisma.config.ts` and requires driver adapters).
3. **Cleaned Up Workspace**:
   - Deleted the custom generated directory `src/generated`.
4. **Added MySQL Driver Adapter (Required in Prisma 7)**:
   - Installed `@prisma/adapter-mariadb` and `mariadb` packages.
   - Updated [database.service.ts](file:///d:/data/Coding%20Cops/nest%20js/restapi/src/database/database.service.ts) to parse `DATABASE_URL` and instantiate `PrismaClient` with the `PrismaMariaDb` driver adapter.
5. **Fixed Syntax & Imports in Services**:
   - In [users.service.ts](file:///d:/data/Coding%20Cops/nest%20js/restapi/src/users/users.service.ts), removed the syntax-breaking `const prisma = new Prisma(),` statement and the unused `Prisma` import.
   - Moved `findAll()` back inside the `UsersService` class boundaries (it was previously defined outside the class closing brace).
   - In [main.ts](file:///d:/data/Coding%20Cops/nest%20js/restapi/src/main.ts), imported `dotenv/config` at the top to ensure environment variables are correctly loaded upon startup.

---

## 3. Why We Changed It

- **Standardization**: Restoring the CommonJS module system aligns the project with standard NestJS CLI architectures, resolving loader mismatch issues.
- **Robustness**: Generating the Prisma Client to `node_modules/@prisma/client` prevents complex pathing problems and is the officially recommended approach.
- **Prisma 7 Compliance**: Version 7 requires the database connection URL to be managed in `prisma.config.ts` and connections to be initialized explicitly through a driver adapter (like `PrismaMariaDb` for MySQL) at runtime.
- **Cleanliness**: Cleaning up the syntax-breaking `new Prisma()` instantiation and moving the function boundaries inside the class resolves compiler checks and makes the codebase fully operational.

## 4. Minimal Example of Using `DatabaseService` (PrismaService)

Below is a minimal example demonstrating how to inject and use `DatabaseService` inside `UsersService` to run queries without writing it directly into your codebase.

```typescript
import { Injectable, NotFoundException } from '@nestjs/common';
import { DatabaseService } from '../database/database.service';

@Injectable()
export class UsersService {
    // 1. Inject the DatabaseService in the constructor
    constructor(private readonly databaseService: DatabaseService) {}

    // Example 1: Fetching all users from the MySQL database
    async findAll(role?: 'INTERN' | 'ASE' | 'SSE') {
        // Query database using the injected service
        return this.databaseService.users.findMany();
    }

    // Example 2: Finding a user by ID
    async findOne(id: number) {
        const user = await this.databaseService.users.findUnique({
            where: { id },
        });
        
        if (!user) {
            throw new NotFoundException(`User with ID ${id} not found`);
        }
        
        return user;
    }
}
```
